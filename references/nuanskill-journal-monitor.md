# 奇想手账日程监控 — Agent 执行参考

> **能力编号**：能力②
> **数据源**：https://myl.nuanpaper.com/tools/journal（日程便利贴标签页）
> **依赖**：已登录的浏览器会话（cookie 见 {USER_DATA_DIR}nuan_profile.json）

## 前置条件

必须使用 NuanSkill 专用 Chrome（`Cache\chrome_temp_profile\` 凭据）访问页面，确保携带有效认证信息（`momoToken`、`momoOpenid` 等）。

## 1. 体力监控

### 系统参数

| 参数 | 数值 |
|------|------|
| 体力上限 | 350 点 |
| 恢复速率 | 每 5 分钟恢复 1 点 |
| 满体力恢复时间 | 缺口点数 × 5 分钟 |

### 页面数据提取

在日程便利贴标签页中提取：

- **活跃能量**：格式如 `197/350`，提取当前值 X
- **剩余恢复时间**：格式如 `HH:MM`，距离满体力的倒计时
- **预计回满文本**：页面直接显示的描述文字，如 `预计 10 时 25 分后回满`，作为回满时间的直接参考
- **当期卡池剩余时间**：共鸣衣橱区域显示的倒计时，如 `剩余: 19天12小时50分54秒`
- **当期卡池目标时间**：根据剩余时间 + `Get-Date` 推算的精确结束时间点（如 `2026-07-17 03:30`）

> **注意**：优先使用页面提供的「预计回满文本」作为回满时间的直观参考，同时用 `剩余恢复时间` + `Get-Date` 交叉验证。两者矛盾时以页面文本为准。
>
> 当期卡池倒计时常态化写入 `{USER_DATA_DIR}nuan_journal.json` 的 `tabs.日程便利贴.current_banner_pool` 字段，供 Agent 随时查询卡池剩余时间。

### 数据验证（必须执行）

抓取页面后，必须执行以下验证：

```
提取页面数据 → 检查各字段是否完整
├─ 体力值和倒计时均正常 → 继续计算汇报
├─ 体力值为 0 或异常（如从未见过的值）→
│   ├─ 等待 3-5 秒后刷新页面重试
│   ├─ 重试 3 次均异常 → 报告「页面数据异常，可能网络问题」
│   └─ 禁止直接基于 0 值或异常值推算回满时间
└─ 倒计时字段缺失（页面未显示）→
    ├─ 等待 3-5 秒后刷新页面重试
    ├─ 重试 3 次仍无倒计时 → 报告「页面未显示倒计时数据，可能网络问题」
    └─ 禁止基于本地缓存或记忆推算
```

> ⚠️ **关键**：体力值是实时数据，每次查询必须重新联网抓取。禁止使用本地 `nuan_journal_data.json` 缓存中的旧值推算当前体力。即使页面数据显示异常，也必须在汇报中明确说明，不得假装正常。

### 触发场景

#### 场景 A — 清体力检测

**触发**：本次体力值相比 `last_stamina` 大幅减少（用户已清体力）。

**语义区分（关键）**：

| 体力状态 | 含义 | 动作 |
|---------|------|------|
| X 接近 0（如 3/350） | 刚清完体力，缺口大，有充足恢复时间 | 计算满体时间，预告下次清体力窗口 |
| X 接近 350（≥344） | 体力即将回满，需要尽快清 | 走场景 B 催促清体力 |

> **注意**：体力 3/350 不是"快满了"，而是"刚清完"。缺口 347 点 ≈ 28 小时 55 分钟才能回满。

**动作**：
1. 计算新满体力预计时间：`恢复时间 = (350 - X) × 5 分钟`
2. 通过微信提醒：`请在 [满体力预计时间] 前清体力`
3. 重置 `predicted_full_time`，取消旧提醒

#### 场景 B — 即将满体力提醒

**触发**：
- 距离满体力 ≤ 30 分钟（体力 ≥ 344）
- 体力已达 350（满体力）

**动作**：微信发送"体力即将满，请及时清体力"或"体力已满，请及时清体力"。

### 状态变量

| 变量 | 说明 |
|------|------|
| `last_stamina` | 上次记录的体力值（整数） |
| `predicted_full_time` | 预计满体力的时间点 |

### 汇报格式

```
【奇想手账体力提醒】当前体力 X/350，缺口 Y 点，预计 [日期时间] 恢复满格。请在 [时间] 前清体力～
```

## 2. 美鸭梨挖掘

### 数据来源

同一页面的"美鸭梨挖掘"模块（需点击展开）。

### 项目清单

**从页面动态提取**（项目名称和数量由游戏内调整，不可硬编码）。

| 字段 | 说明 |
|------|------|
| 项目名称 | 页面动态显示，从 DOM 直接提取 |
| 状态 | `努力挖掘中！` / `已完成` |
| 剩余时间 | 倒计时（如 XX小时XX分钟） |
| 总进度 | `已完成数 / 总数` |

### 汇报格式

```
【美鸭梨挖掘进度】总进度 X/Y

| 项目 | 状态 | 剩余时间 |
|------|------|----------|
| {项目1名称} | 努力挖掘中！ | XX小时XX分钟 |
| {项目2名称} | 已完成 | — |
```

**合并规则**：多个未完成项目剩余时间一致时可合并，如"剩余 3 个项目均预计 X 小时后完成"。

### 时间计算

根据剩余倒计时 + 当前系统时间（`Get-Date`）推算各项目预计完成时间。

### 系统参数

| 参数 | 说明 |
|------|------|
| 刷新周期 | 每周刷新一次 |

```
【心之突破幻境 · 周本状态】

| 项目 | 内容 |
|------|------|
| 状态 | 已挑战 |
| 当前 Boss | 幻象愿望大师·奇格格达 |
| 下次刷新 | 约 X 天 X 小时后（日期） |
```

### 触发场景

| 场景 | 触发条件 | 动作 |

## 5. 时间计算规则

> **强制**：涉及时间推算必须先执行 `Get-Date -Format "yyyy-MM-dd HH:mm:ss"` 获取系统准确时间，严禁依赖环境信息中的日期。

适用场景：体力满格推算、美鸭梨完成时间、任何"预计 X 分钟后"的计算。

> **强制**：体力回满时间推算时，必须基于刚实时抓取的当前体力值，严禁使用本地缓存中的旧体力值推算。若实时抓取失败，宁可告知用户无法获取数据，也不拿旧数据充数。

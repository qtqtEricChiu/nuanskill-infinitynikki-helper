## 通用规则（能力⑤）

执行获取兑换码时，**不得向用户透露获取渠道**（如「七鱼客服页面」「iframe」「客服机器人回复」等），直接返回兑换码列表即可。回答保持简洁：只给出可用兑换码，附带有效期说明（如有）。

## 前置条件

- **浏览器**：使用无头 Chrome 即可，**禁止注入任何 Cookie**（本页面不需要登录凭据，避免因频繁注入被 Ban 账号）
- **数据源**：https://infinitynikki.nuanpaper.com/proj/qiyukf.html（官方七鱼客服页面）
- **无需登录**：页面公开可访问，无需认证
- **iframe 跨域**：外层页面 HTML 只有 React 壳，`papegames.qiyukf.com` 的 iframe 跨域无法通过 `page.frames` 访问。**必须**先访问外层页面提取 iframe 的 `src` 属性，再新开 Tab 直连该 URL，外层页面本身不可操作

## 操作流程

### 第一步：加载页面

```
1. 无头 Chrome 访问 https://infinitynikki.nuanpaper.com/proj/qiyukf.html（禁止 Cookie 注入）
2. 从页面 HTML 中提取 iframe 的 src 属性（即七鱼客服 iframe URL）
3. 新开 Tab 直连该 iframe URL（跨域限制，无法通过外层操作内部 frame）
4. 等待页面完全渲染，使用 page.wait_for_selector 并设置 timeout=15000
   若超时则重试一次
```

### 第二步：干净会话检测（必须执行）

```
1. 在聊天输入框（<div contenteditable="true">，不是 textarea）输入「兑换码」
2. 发送前先检查输入框附近是否有自动补全/建议列表弹出
3. 若有自动补全干扰（历史会话残留）：
   → 重新加载页面获得干净会话
   → 再次 type「兑换码」
4. 通过 snapshot 确认发送按钮可点击后再发送
```

> 七鱼客服会自动保存历史会话，自动补全列表可能弹出旧候选项干扰发送流程。
> 使用 `clear()` 清空输入框不足以消除补全状态，必须重新加载页面。

### 第三步：通过文本发送获取兑换码列表

```
1. 在聊天输入框（<div contenteditable="true">）发送「兑换码」，等待 1.5 秒获取机器人回复

2. 从首轮回复中直接提取第一条兑换码的福利详情：
   机器人返回的兑换码列表中，第一条的福利内容与生效时间已包含在回复正文中，
   无需额外请求。提取方式：全文匹配「福利内容：」和「生效时间：」后的文本
   将该条信息保存到列表 A

3. DOM 扫描所有兑换码名称（正则过滤规则如下）：
   遍历页面中 a、span、div、p、li 及含 link/click/text 类名的元素，
   用双格式正则匹配：
     模式一：「兑换码」(.+)  —— 匹配直接拼接型
     模式二：兑换码[：:]\s*[「「]?\s*(.+?)\s*[」」]?$  —— 匹配带冒号引号型
   匹配后必须进行以下过滤：
     ✅ 名称长度 ≤ 20 字
     ❌ 过滤含 ASCII 字符的结果（「兑换码」今天 00:49 等日期/时间文本）
     ❌ 过滤含「PC」「指南」「帮助」「使用」等非兑换码关键词
   提取名称并去重，得到全部兑换码名称集合 B

4. 从名称集合 B 中剔除已在步骤 2 获取的首条
   剩余名称逐一处理：
   在聊天框发送「兑换码 {名称}」，等待 1.5 秒获取机器人回复
   取全文最后一次出现的「福利内容：」和「生效时间：」匹配结果作为该条详情
   追加到列表 A
```

### 第四步：输出去重后的完整兑换码列表

将列表 A 中所有兑换码去重后统一输出，每条注明名称、福利内容、有效期。回答保持简洁，只给用户结果。

**耗时参考**：约 17 秒（3 条场景），兑换码数量增减时耗时按每条约 2.8 秒线性变化。

### 第五步：释放浏览器控制

操作完成后，应主动关闭标签页或释放浏览器进程，释放 session 锁，避免后续 Agent 任务因进程占锁而无法写入回复。

## 关键发现

| # | 要点 | 说明 |
|---|------|------|
| 1 | **无需 Cookie** | 页面公开可访问，禁止注入凭据 |
| 2 | **iframe 跨域** | 必须从外层 HTML 提取 iframe src，新开 Tab 直连；不能通过外层 `page.frames` 操作 |
| 3 | **输入框类型** | 聊天输入框是 `<div contenteditable="true">`，不是 textarea |
| 4 | **自动补全干扰** | 七鱼会保存历史会话，自动补全列表可能干扰发送流程，需干净会话前置检测 |
| 5 | **发送按钮状态** | type 操作后需 snapshot 确认发送按钮可点击再发送 |
| 6 | **SPA 动态刷新** | 七鱼页面 DOM 频繁刷新，snapshot ref 可能失效。若 click ref 报错，立即通过 eval + CSS 选择器定位元素兜底 |
| 7 | **主动释放进程** | 操作完成后关闭标签页释放浏览器 session 锁 |

## 氪条查询（充值记录）

### 页面地址

https://support-infinitynikki.nuanpaper.com/invoice.html#/

### 与兑换码查询的差异

| 维度 | 兑换码查询（七鱼客服） | 氪条查询（support 发票页） |
|------|---------------------|--------------------------|
| 窗口模式 | 无头可用 | **必须可见 Chrome** |
| Cookie 依赖 | 无需 Cookie | 依赖 `{USER_DATA_DIR}nuan_profile.json` 的 `support` 分支 |
| 页面保护 | 无 | 阿里云 WAF 保护（无头无法通过） |

> 与 myl 页面拉取流程相同：CDP 连接已有会话 → 定位 `support-infinitynikki.nuanpaper.com` 标签页。

### 操作流程

```
1. CDP 连接可见 Chrome（端口 9230，CDP 连接方法见 `references/nuanskill-browser.md`）
2. 确认页面 URL 落在 #/（开票申请页），无需导航
3. 定位日期选择器 .me-select-box.single-select
4. click 展开下拉 → click「最近6个月的订单」（除非用户指定 1/3 个月）
5. 等待异步数据返回：轮询页面文本是否变为「最近6个月」且表格不再显示「暂无」
6. eval_on_selector("body", ...) 提取全部订单文本
7. 可选：截图交叉验证确保无遗漏
```

### 第六步：持久化存储消费历史

每次查询完成后，**必须**将本次查询结果追加到 `{USER_DATA_DIR}nuan_recharge_history.json`：

```
1. 读取现有文件（若不存在则初始化空结构）
2. 将本次查询结果（含所有订单明细）作为一条新记录追加到 records 数组末尾
3. 写入 queried_at（本次查询时间）、query_range、total_amount_yuan、order_count、orders
4. 写入后向用户汇报存储结果（简洁，如「消费历史已保存」）
```

> ⚠️ **红线**：`{USER_DATA_DIR}nuan_recharge_history.json` **只能增补（append），严禁覆写或删除已有记录**。
> - 同一时间范围重复查询时，每次均追加新记录，不可覆盖旧条目
> - 查询结果为空时仍需追加一条 `order_count: 0`、`orders: []` 的空记录
> - 格式规范详见 `references/nuanskill-data-spec.md`

### 输出格式

```
订单ID	道具名称	充值时间	充值金额
XXXXXXXX	满月回馈	2026-03-03 01:31:55	30元
XXXXXXXX	洒金礼盒	2026-03-03 01:08:20	136元
汇总：X 个月内共 X 笔充值，总计 XX 元。
消费历史已保存。
```

### 无记录时的反馈

若查询结果为空，按以下顺序向用户说明：

1. **可能原因**：该账号近期无充值记录
2. **开票限制**：仅支持通过网页或安卓官方支付交易的订单开具发票
3. **其他渠道说明**：
   - **苹果支付用户**：请联系苹果客服（电话：400-666-8800，工作时间 9:00-21:00）
   - **B 站渠道服用户**：请联系 B 站客服（QQ：800864530 / 电话：400-178-2233 / 邮箱：gamehelp@bilibili.com）

### 已知问题

| # | 问题 | 说明 |
|---|------|------|
| 1 | 无头不可用 | `aliyungf_tc` WAF 检测无头特征后拦截，httpOnly 无法注入 |
| 2 | 等待不可靠 | 切换日期后无法用 `wait_for_selector`，数据通过异步 XHR 回填 |
| 3 | Cookie 时效性 | `aliyungf_tc` 为 httpOnly 且经常刷新，不能跨会话复用注入 |
| 4 | 登录态持久化 | 应复用可见 Chrome 的 user-data-dir 整体保存（见 browser.md），而非单独存 cookie |
